# Dark Factory Toolkit — Foundation Design

> Output of the initial design session conducted against
> [`docs/dark-factory-toolkit.md`](../dark-factory-toolkit.md).
> This document captures the architectural decisions that frame v1.
> It is descriptive, not prescriptive about line-level implementation.

## 1. Purpose & character

**dft** (Dark Factory Toolkit) is a **headless workflow engine** for
spec-driven software production. It accepts a description of what a
human wants built — a *demand-package* — and drives that work through a
deterministic pipeline of agent invocations, tool runs, and gates until
the work is either accepted by an adversarial evaluator or remediated.

dft is **not** an outer-shell personal agent (à la OpenClaw / Hermes).
It is a **process supervisor** for LLM and tool work, in the spirit of
Temporal/Dagster/Argo, but specialized for software production with
spec-kit-style methodology at its core. A chat surface, if ever wanted,
is a *consumer* of the engine, not part of it.

It is **opinionated about three things**:

1. **Intent / Design / Orchestration** are the three phases of every
   submission.
2. **Build and Eval are adversarial.** A test plan generated by an
   agent — distinct from the one that built the artifact — gates every
   submission.
3. **The Work Breakdown Structure is the spine.** It is authored by an
   agent, versioned in git, amendable by the Fix-Planner, and every
   piece of agent work is keyed to a spec inside it.

## 2. Vocabulary

| Term | Meaning |
|---|---|
| **Demand-package** | Output of Intent. A description plus acceptance criteria. **One demand-package = one unit of progress = one Eval gate.** |
| **Increment** | The git-backed execution envelope for one demand-package: one increment branch, many spec branches, one final merge to the repo default branch. |
| **Spec** | Unit of *work* a single coding-agent invocation can take on. The unit spec-kit operates on. |
| **WBS** | The DAG of specs belonging to one demand-package. Authored by an agent in Solution Design. Append-only. |
| **Lane** | A flow template (DSL file) that knows how to execute one spec. Assigned **per spec** by the Lane Selector. |
| **Flow** | A sequenced list of steps written in the dft DSL. |
| **Step** | Atomic unit inside a flow: `agent`, `tool`, `function`, or `workflow`. |
| **Phase flow** | A flow the engine invokes at a fixed point in the macro-loop (`intent`, `design`, `eval`, `fix-planner`). |
| **Run** | One execution attempt of a flow, identified by a run-id. |
| **Job** | One submission of a demand-package through the full pipeline. |

## 3. The macro-loop

The engine's top-level orchestration is **engine code, not a flow**.
It enumerates *what units of work exist and in what order*; flows
describe *how to do each unit of work*.

```
dft submit <demand>
  │
  ├─ Intent phase
  │     run flows.intent  →  demand-package
  │
  ├─ Increment setup
  │     discover repo default branch
  │     create increment branch from default branch HEAD
  │
  ├─ Solution Design phase
  │     run flows.design   →  architecture blueprint
  │                          WBS (DAG of specs)
  │                          lane assignment per spec
  │                          [optional manual gate]
  │
  ├─ Orchestration phase
  │     for each spec in WBS (sequential v1; worktree-per-spec):
  │         create spec branch/worktree from current increment branch
  │         run the spec's lane flow
  │         commit step-by-step (one step = one commit)
  │         merge spec branch back to increment branch on success
  │
  ├─ Eval phase
  │     run flows.eval-plan-author  →  test plan
  │     execute test plan deterministically
  │                                   →  verdict (pass | fail + findings)
  │
  ├─ Final review / merge phase (only after pass)
  │     review increment diff against default branch
  │     merge increment branch back to default branch
  │
  └─ if fail:
        run flows.fix-planner
          option (a) → WBS amendment (append specs, re-Build, re-Eval)
          option (b) → child demand-package (recurse pipeline)
        bounded retry; escalate to inbox_entries when exhausted.
```

## 4. Architectural decisions (one section per)

### 4.1 Identity — headless workflow engine

Engine invoked as `dft submit | status | inspect | cancel | resume`.
No top-level agent loop. No built-in chat surface. State is durable on
disk; the engine can crash and resume.

### 4.2 Build/Eval — generated-then-static, gated at demand-package boundary

- **Posture:** gate at the end of the demand-package. Not co-pilot,
  not controller.
- **Shape:** an agent reasons about testable surfaces and emits a
  deterministic test plan; the engine executes it.
- **Remediation on failure:** the Fix-Planner agent emits either
  (a) a WBS amendment or (b) a child demand-package. Both bounded;
  escalation surfaces to `inbox_entries`.

### 4.3 Storage — α (git-native + sqlite), behind ports

- v1 storage: artifacts live as files in the target repo under
  `.dft/`, audited by git commits. Engine state (run status, queue,
  retries) lives in `.dft/state.db` (sqlite).
- All storage access goes through ports — `ArtifactRepository`,
  `JobStore`, `RunStore`. The engine never touches git or the
  filesystem directly. The day α's parallel ceiling bites, the storage
  impl swaps without engine changes.
- **Artifact location:** sidecar `.dft/` directory inside the target
  repo (option i). v1 does not support multi-repo orchestration.
- **One step = one commit, hard invariant.** The engine owns all
  commits. Agents and tools must leave changes unstaged; they never
  commit. Commits carry trailer metadata (run-id, spec-id, step-id,
  upstream refs). Resume-from-failure restores the worktree to the
  last committed step.
- **Remote-only steps produce zero commits.** Steps that mutate only
  remote state (`gh pr create`, `gh pr merge`, `git push`) are declared
  remote-only and skip the commit boundary. Their audit lives in
  RunStore plus `.dft/runs/<run-id>/<step-id>/`. The "one step = one
  commit" rule is strictly "every step that mutates local state
  produces exactly one commit."

#### 4.3.1 Branch topology — increment branch owns integration

- **Default branch is the release baseline.** At the start of each
  increment, dft resolves the repository's default branch, updates or
  verifies that ref, and creates a new increment branch from the default
  branch HEAD. No spec work is based directly on the default branch.
- **Spec branches are children of the increment branch.** For each spec,
  dft creates a per-spec branch and worktree from the current increment
  branch HEAD. In v1 specs run sequentially, so each later spec naturally
  sees the prior successful specs after they have been merged back into
  the increment branch; the same topology supports parallel spec
  worktrees later.
- **Spec Kit branch integration is current-HEAD based.** The
  `speckit.specify` agent delegates branch creation to the mandatory
  `before_specify` hook (`speckit.git.feature`). That git extension's
  branch script creates the branch with `git checkout -b "$BRANCH_NAME"`
  from the current HEAD. Therefore dft must invoke specify from a
  worktree already checked out to the intended increment base.
- **dft owns branch names.** When invoking the Spec Kit branch hook, dft
  passes `GIT_BRANCH_NAME` so the hook uses the exact per-spec branch
  name instead of generating one. dft also passes or records the exact
  `SPECIFY_FEATURE_DIRECTORY` because the specify agent treats the spec
  directory and git branch name as independent.
- **Merges are staged.** Successful spec branches merge back only to the
  increment branch. The increment branch merges back to the default
  branch only after all specs complete, Eval passes, and the final code
  review gate approves the full increment diff.

### 4.4 Lanes & spec-kit relationship

- The `spec` lane is itself a flow written in dft's DSL. Its steps
  invoke copilot custom-agents that mirror spec-kit's prompts.
  dft owns orchestration end-to-end; spec-kit's prompt content is
  borrowed where useful.
- **Lane catalog**:
  - **MVP:** `spec`, `streamlined`, `manual`.
  - **Post-MVP:** `recursive` (spec-level iterate-until-converged),
    `agentic`.
- **Catalog shape:** open plugin. Lanes are flow files; built-ins are
  embedded in the binary, user lanes live in `.dft/lanes/`. User wins
  on name conflict.
- **Lane assignment:** per-spec. The Lane Selector (an agent step in
  the design flow) emits one lane per spec; static for the run.

### 4.5 Agent-invocation contract

| Sub-decision | v1 |
|---|---|
| Adapter shape | Adapter posture; v1 ships `copilot` adapter + generic `command` fallback. `claude`/`codex`/`aider`/`mcp` are post-v1. |
| Audit capture | Always-on; artifacts (git diff), full transcript, structured output files all written to `.dft/runs/<run-id>/<step-id>/`. |
| Downstream consumption | `capture: true` promotes stdout to `{{ steps.<slug>.stdout }}`. Combine with `export_as: <name>` to also bind the captured value to a named flow var usable as `{{ <name> }}` across stages. |
| Templating language | Jinja-style. Named context: `spec`, `demand_package`, `project`, `steps.<slug>.*`, `vars.*`, `run`, `env`. Filters include `from_json`, `from_yaml`, `default`. |
| Structured data protocol | Stdout, parsed by the consumer (`{{ … \| from_yaml }}`). Subsumes write-to-file conventions. |
| Error semantics | `on_error: continue \| fail \| retry(n, backoff) \| escalate`; default `fail`. `escalate` surfaces to `inbox_entries` and blocks the spec. |
| Timeouts | `timeout:` settable at flow / stage / step level; inner overrides outer. Step timeout enforced by the runner via subprocess kill. |
| Per-step selection | Each `agent` step picks adapter, agent file, and model. |
| Commit posture | Agents and tools leave changes unstaged. The engine commits at step boundary (see §4.3). |

### 4.6 Concurrency

| Axis | v1 | Notes |
|---|---|---|
| 6a Step parallelism within a flow | Sequential only | Most flows are short and linear. |
| 6b Spec parallelism within a demand-package | Sequential (cap=1), **but branch/worktree-per-spec from day one** | Spec branches are based on the increment branch; day we flip the cap to N, nothing else changes. |
| 6c Multi-job (across submissions) | Single-job + FIFO queue | Multi-job lands with post-α storage. |
| 6d Cancellation | SIGTERM → grace → SIGKILL; run state = `cancelled`; artifacts kept | No automatic worktree rollback. |
| 6e Restart-from-failure | One step = at most one commit, with metadata trailers | Crash recovery scans recent commits and reconciles RunStore. |

### 4.7 Implementation language — Go

Engine dependency surface stays small: stdlib + sqlite + git porcelain
+ templating + MCP client (post-v1). **All model calls go through
agent CLI subprocesses**, including Eval's LLM-judge steps. The engine
never embeds an LLM SDK.

### 4.8 Shared context, target env, provisioning

- **Project context (posture B, implicit):** every `agent` step
  auto-receives the project context bundle from `.dft/context/`
  (constitution, glossary, ADRs). Steps may opt out. Run audit
  records content hashes of attached files for replay determinism.
- **Target environment:** v1 targets **GitHub Copilot CLI**
  exclusively. "Agent" in the DSL means a copilot custom-agent
  invocation (markdown agent file in the repo).
- **Provisioning:** dft lays down required assets in the target repo:
  - spec-kit's agent files and templates,
  - dft-provided custom agents (code-review, merge, dft-flavored
    `specify`, WBS-builder, Lane Selector, Eval test-plan-author,
    Fix-Planner),
  - `.dft/` scaffold: `context/`, `lanes/`, `flows/`, `runs/`,
    `state.db`,
  - starter constitution.
  Commands: `dft init`, `dft sync` (update in-repo assets).
  Each provisioned agent file carries a frontmatter version header;
  audit records content hashes so replay survives upgrades.
- **Skills (Hermes/agentskills.io):** not in v1. Lanes are dft's
  equivalent of procedural memory.
- **Cross-run memory:** `.dft/runs/` plus git history is sufficient
  for v1.

### 4.9 DSL

Top-level shape (illustrative, not normative):

```yaml
flow: spec.lane
target: spec               # which lane category this flow serves
description: |
  Drives a spec through spec-kit-style /plan → /tasks → /implement.

vars:
  copilot_flags: "--allow-all -s --no-ask-user --autopilot"

defaults:
  timeout: "600s"
  max_retries: 1

inputs:
  spec:           { type: SpecRef, required: true }
  demand_package: { type: Ref,     required: true }

outputs:
  artifacts:        { type: [ArtifactRef] }
  wbs_amendments:   { type: [WBSAmendment], optional: true }

stages:
  plan:
    timeout: "600s"
    steps:
      - id: plan
        type: agent
        adapter: copilot
        agent: speckit-plan
        model: ${defaults.model}
        prompt: |
          Plan the implementation of:
          {{ spec.description }}

          Acceptance criteria:
          {% for c in demand_package.acceptance_criteria %}
          - {{ c }}
          {% endfor %}
        capture: true
        export_as: plan_output
        on_error: fail
    verify:
      checks:
        - fn: file_exists
          args: ["specs/{{ spec.id }}/plan.md"]

  implement:
    timeout: "1800s"
    steps:
      - id: implement
        type: agent
        adapter: copilot
        agent: speckit-implement
        prompt: |
          Implement the following tasks:
          {{ steps.tasks.stdout }}
        on_error: retry(n: 2, backoff: 30s)
        verify:
          checks:
            - fn: command_exit_zero
              args: ["go", "build", "./..."]
    after:
      - id: push
        type: function
        fn: git_push
        args: { remote: "origin", branch: "{{ spec.branch }}" }
```

DSL elements:

- **Stages (optional).** A flow may group its steps into named
  `stages:`, each with its own `before:` / `after:` / `verify:` /
  `timeout:`. Stages are sequential. Most simple flows skip stages
  and use a flat `steps:` list. Long lifecycle flows (e.g., `spec.lane`)
  benefit from grouping.
- **Stage `before:` / `after:` hooks.** Lists of step-shaped entries.
  `before:` runs before the stage's main steps; `after:` runs **only
  if** the stage's main steps all succeeded. Stage failure is a
  terminal event — control returns to the engine's macro-loop, which
  routes to Fix-Planner. There is no `on_failure:` block.
- **Step types:** `agent`, `tool`, `function`, `workflow`. No raw shell.
- **`tool` steps** invoke external programs by **argv only**
  (`command: ["git", "push", "origin", "main"]`). No shell
  interpretation, no pipes, no redirects. Stdin / env / cwd
  configurable on the step.
- **`function` step closed set** (v1): `wait_for_human`, `set_var`,
  `branch` (create/switch from an explicit base), `merge`,
  `commit_message` (compose a commit message via an agent without
  committing), `git_push`, `git_branch_current`, `git_default_branch`,
  `gh_pr_create`, `gh_pr_number_for_branch`, `gh_pr_merge`,
  `gh_pr_wait_checks`. The engine implements each; flows declare
  arguments declaratively. The closed set grows as patterns repeat.
- **`workflow` step:** invokes another flow by name, passing typed
  inputs.
- **`loop` step:** bounded iteration over a sub-sequence of steps.
  Requires `max_iterations: <int>` (hard cap, no exceptions) and
  `exit_when:` (early-exit predicate from a closed set:
  `exit_code: 0`, `no_critical_findings: true`,
  `file_exists: <path>`, `check_passes: <verify-ref>`). Loops are
  permitted only inside lanes that explicitly need convergence
  (`review`, post-v1 `recursive`); the engine validates this at
  submit-time.
- **Per-step `verify:`** runs after the step's main work. Shape:
  `verify: { checks: [{fn, args}] }`. Closed `fn` set:
  `file_exists`, `file_missing`, `command_exit_zero`,
  `grep_matches`, `count_matches_at_least`, `json_path_equals`, `os`
  (single command, exit-code is the predicate value — constrained
  escape hatch, no piping into other commands). Verify checks are
  predicates, not steps: they produce pass/fail, no commit, no
  per-check audit step. Failures are reported structurally so the
  Fix-Planner can consume them as JSON.
- **Pre-hook on a step:** `setup:` (list of step-shaped entries that
  run before the main work).
- **Templating:** Jinja-style. Built-in filters: `from_json`,
  `from_yaml`, `default`, `length`, `join`. Conditionals and loops
  supported in templates. No arbitrary code execution.
- **Timeouts.** `timeout:` may appear at flow, stage, and step level.
  Inner overrides outer. The flow runner enforces step timeouts via
  subprocess kill (`SIGTERM` → grace → `SIGKILL`).
- **`vars:` and `defaults:` blocks.** Flow-local constants live in
  `vars:` (referenced as `{{ vars.<name> }}`). Inheritable defaults
  for `timeout` and `max_retries` live in `defaults:`. CLI flags and
  env overrides apply to both.

What the DSL deliberately does **not** have:

- Raw shell steps. If a flow needs an operation that isn't expressible
  as `agent` / `tool` (argv) / `function` (closed) / `workflow`, the
  fix is to grow the `function` closed set, not to reach for bash.
- `on_failure:` hooks. Failure handling is the engine's job.
- Unbounded loops or general-purpose control flow. Loops require
  `max_iterations` and a closed-set `exit_when` predicate.
- Per-step commit toggles. The engine commits at every local-mutating
  step boundary; flows cannot opt out.

### 4.10 Flow I/O typing

Flows declare `inputs:` and `outputs:` with types drawn from a small
closed set. JSON Schema for each type ships embedded in dft.

| Type | Carries |
|---|---|
| `SpecRef` | Identity + description + acceptance criteria of one spec |
| `Ref` | A reference to any addressable artifact in the run |
| `ArtifactRef` | A file (or set of files) the engine should track |
| `Verdict` | `{ status: pass\|fail, findings: [...] }` |
| `WBSAmendment` | `{ specs: [SpecDef], rationale: string }` |
| `String`, `Json`, `Yaml` | Scalar/structured stdout passthroughs |

Engine validates inputs/outputs at submit-time. Validation failures
abort the run before any LLM call is made.

## 6. Engine internals

### 6.1 Major components

- **CLI** — `dft submit | status | inspect | cancel | resume | init | sync`.
- **Macro-orchestrator** — implements the Intent → Design → Orchestration → Eval → Fix-Planner loop in Go.
- **Flow runner** — loads a flow file, validates I/O, executes steps sequentially, handles `setup` / `verification` / `on_error`.
- **Adapter registry** — `copilot` and `command` adapters for v1; port for future ones.
- **Template engine** — Jinja-style renderer with the named context.
- **ArtifactRepository** (port) — git-native impl in v1.
- **RunStore / JobStore** (port) — sqlite impl in v1.
- **Worktree manager** — increment/spec branch creation, final merge
  choreography, and `git worktree` lifecycle for per-spec isolation.
- **Inbox** — surface for `escalate` / manual gates; backed by the `inbox_entries` table.

### 6.2 Crash-recovery contract

1. On every step completion, the engine commits the worktree (if any artifacts), then updates RunStore.
2. On crash:
   - Scan the most recent commits per active worktree.
   - For each commit whose metadata identifies a step the RunStore doesn't know is complete, mark that step complete in RunStore.
   - Resume from the first step that has neither a committing nor a RunStore record of completion.
3. RunStore is the source of truth for "what runs are alive"; commits are the source of truth for "what work has been persisted."

## 7. What is explicitly out of scope for v1

- Multi-CLI support (only `copilot` ships).
- Multi-repo orchestration.
- Multi-job concurrency (single-job + FIFO).
- Step parallelism within a flow.
- Spec parallelism within a demand-package (designed for, not enabled).
- `recursive` and `agentic` lanes.
- General-purpose / unbounded DSL loops. (Bounded loops with a closed
  `exit_when` predicate set are in, scoped to lanes that need
  convergence.)
- Skills / agentskills.io compatibility.
- Cross-run learned memory.
- Chat surface / messaging gateways.
- Non-local execution engines (Docker, K8s, GH Actions).

## 8. Glossary of reference projects

- **spec-kit** — methodology dft mirrors. v1 invokes copilot custom-agents whose prompts replicate spec-kit's slash-command shapes; we do not import spec-kit's machinery as a black box.
- **skrunner** — author's prior Go-based orchestrator of spec-kit flows; validated language choice and broad shape.
- **OpenClaw** — outer-shell personal-assistant agent. Wrong shape for dft, but its file-first artifact instinct is borrowed.
- **Hermes (Nous Research)** — outer-shell agent with pluggable executors, MCP, skills/memory. dft borrows the *primitive ideas* (pluggable adapters, context files, MCP) without the long-lived-agent runtime.

## 9. Open work for the implementation phase

These were deliberately deferred from foundation-level design:

- Concrete JSON Schema definitions for the flow I/O types.
- The full enumerated set of `function` step kinds and `verify.checks`
  `fn` kinds (and their argument schemas). v1 ships an initial set;
  growth policy is "add when a pattern repeats."
- The exact frontmatter format for provisioned agent files (and the
  `dft sync` reconciliation algorithm).
- The starter constitution template.
- The full default contents of each MVP lane and each phase flow.
- The `inbox_entries` schema (currently a placeholder table).
- Logging / tracing / observability strategy (probably structured
  JSON to `.dft/runs/<run-id>/engine.log` plus a `dft inspect` UI
  over it).
- Versioning strategy for dft itself and how `dft sync` upgrades
  in-repo assets safely.
